How the loop stays honest

PHASE 1 — INVESTIGATE

Tools first, opinions later

The model starts with only the diagnostic tools — summaries of dmesg fault classes, the BMC event log, and per-device AER register decodes. It cannot conclude anything it hasn't queried.

PHASE 2 — SUBMIT

A verdict is a tool call

Only after investigating is submit_rca_report exposed. The tool validates the submission — empty root cause or unknown category gets REJECTED back in-context, forcing the model to retry with real arguments.

FALLBACK

Always ship a report

A rule-based scorer runs the same evidence independently: it cross-checks every agent verdict (irreconcilable component disputes are settled by the evidence, recorded in the report), and runs standalone when no LLM is reachable (--no-llm). The pipeline degrades gracefully, never silently.

Native diagnostic tools (C#, unit-tested)

redfish.*

DMTF LogEntry parser — severity rollups, per-component event queries

pcie.*

AER status registers decoded to named spec bits; link downtrain detection

dmesg.*

Kernel log classified: EDAC CE/UE, MCE, NVMe, AER, enumeration, offload, I/O, thermal

boot.*

Pre-OS boot flow: POST codes across SEC/PEI/DXE/BDS, stage errors, OS-handoff status

dpu.*

DPU-internal (ARM-side) console: ATF/UEFI boot chain, NIC firmware, DOCA flow engine

LLM endpoint — swappable via config

Ollama (default, local llama3.2) · OpenAI · Azure OpenAI

One OpenAI-compatible connector covers all three; provider and model switch via appsettings.json, RCA_* environment variables, or CLI flags.
